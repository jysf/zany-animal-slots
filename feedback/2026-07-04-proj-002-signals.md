  - id: adversarial-mutation-must-be-behavior-distinguishing
    type: lesson
    summary: "An adversarial guard-mutation only has teeth if some observation can distinguish the mutation from the original. Three sub-cases: (1) mutating code redundant with a language guarantee (e.g. ES2019 stable sort) is a no-op no test can catch — document, don't 'fix'; (2) a real behavioral difference the LITERAL test text doesn't yet pin — add the pinning assertion; (3) a difference not observable via RETURN VALUE under current data (e.g. a reactive seam with one registered item) — pin it structurally with a delegation/spy test. For a 'no-op seam' spec, author the teeth test in Failing Tests up front."
    evidence: "PROJ-002 STAGE-008 SPEC-045 + SPEC-048 + SPEC-049 (N=3). SPEC-045 (case 1): removing buildStrip's `|| a.ord - b.ord` tie-break failed 0/320 tests — Array.sort is stable (ES2019+) and items are inserted in `symbols` order, so the tie-break is provably redundant; kept it (defensive) + documented. SPEC-048 (case 2): the design named three mutations but two (getChannel's active-gain read; applyMix's active-bed restore) weren't distinguished by the literal test text — the build agent added one assertion each so the mutations break a test. SPEC-049 (case 3): mutation (c) 'getActiveMachine ignores storage, returns const default' broke no test, because the persisted-id fixture ('wild-and-whimsical') EQUALS DEFAULT_MACHINE_ID in the single-machine registry — no return-value assertion can distinguish read-storage from const-default until a 2nd machine exists (SPEC-051). Verify closed it with a `vi.spyOn` delegation test ('getActiveMachine delegates to readActiveMachineId') that passes on correct code and fails under the mutation. Fix across all three: before prescribing 'revert X → test Y fails', confirm SOME observation distinguishes X — a pinned value (case 2) or a structural spy (case 3) — and author that test first; if none can (case 1), it's redundant code, not a test gap."
    status: noted
    disposition_at: stage-close
    first_flagged: 2026-07-05
    notes: >-
      Refines the STAGE-007 "revert-and-confirm-the-guard-fails" default verify move: before
      prescribing (or trusting) an adversarial mutation, confirm the mutated line is
      behavior-distinguishing — i.e. there EXISTS an input for which the original and mutated
      code produce different observable output. If no such input can be constructed, the code is
      provably redundant (often with a language/library guarantee), which is a code-quality note
      (document or delete the redundancy), NOT a test-strength defect. Practical test for the
      designer: when writing a "mutate X and confirm a test fails" check into a spec's Notes,
      sanity-run the mutation once during design so the spec never prescribes a no-op probe. Also
      a template angle: verify agents should distinguish "guard has no teeth (add a test)" from
      "mutation has no observable effect (redundant code)" — different remedies.

## meta

**Multi-project is the trigger.** Every PROJ-001 resolution bug was masked while only
one real project existed. The moment PROJ-002 opened as a second, *active* wave beside a
*shipped* PROJ-001, the status-blind resolver started pointing every default-scoped
command at the wrong (finished) project. This is the concrete cost of the resolution
cluster the PROJ-001 retrospective flagged: it only bites once a repo actually holds
more than one project — exactly the state a "continuous, multi-wave" repo is designed to
reach. The orchestrator worked around it all session with explicit `ACTIVE_PROJECT=…` /
`PROJ-002` args, but a fresh-session user following the template verbatim would read
stale PROJ-001 status and a green cost-audit that never even inspected the active wave.
